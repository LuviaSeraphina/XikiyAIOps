## 2. 核心性能指标

### 2.1 安全护栏延迟

安全护栏是请求链路上的第一道关口，其延迟直接影响端到端响应时间。

| 测试项 | 最小 | 最大 | 平均 | 单位 | 判定 |
|--------|:---:|:---:|:---:|------|:--:|
| 意图分类 (4 层递进) | 0.005 | 0.034 | **0.016** | ms | ✅ 亚毫秒 |
| 注入检测 (7 层) | 0.007 | 0.032 | **0.013** | ms | ✅ 亚毫秒 |
| 命令白名单校验 | 0.007 | 0.034 | **0.018** | ms | ✅ 亚毫秒 |
| 安全流水线总延迟 | 0.005 | 0.086 | **0.026** | ms | ✅ 亚毫秒 |

> 结论: 安全护栏对请求处理无感知延迟，四道检测累计 < 0.1 ms。

### 2.2 MCP Tool 执行延迟

| Tool | 延迟 | 单位 | 说明 |
|------|:---:|------|------|
| `system_load` | < 0.3 | ms | 热启动，纯内存读取 |
| `memory_info` | < 0.3 | ms | psutil 内存快照 |
| `system_info` | < 0.5 | ms | 含平台检测 |
| `process_detail(1)` | 32.8 | ms | 含模块冷导入 |
| `process_tree` | 34.4 | ms | 466 进程树构建 |
| `process_inspect` | 587 | ms | 含 500ms CPU 预热采样 |
| `security_auth_failures(24h)` | 493 | ms | journalctl 查询 |
| `security_suid_scan` | 69.3 | ms | find 扫描 /usr/bin |
| `memory_oom_history(24h)` | 243 | ms | journalctl -k |
| `disk_large_files(/, 10, 100MB)` | 14,133 | ms | find 扫描全盘，受磁盘性能影响 |

> `process_inspect` 的 500ms 为 CPU 采样预热 (`time.sleep(0.5)`)，属于设计补偿而非性能瓶颈。
> `disk_large_files` 延迟取决于磁盘 I/O 性能和文件数量。

## 5. 端到端延迟分解

以一次完整对话 "查看系统负载" 为例，延迟构成：

```
安全前置检查        0.026 ms  ▏ 可忽略
LLM 推理          取决于 Provider 和模型大小
  ├─ Ollama + Qwen3 4B (本地)    ~2000 ms
  └─ DeepSeek API (远程)         ~500 ms
Tool 调用 (system_load)    < 0.3 ms  ▏
SSE 推送首个 token          < 10 ms  ▏
──────────────────────────────────
总延迟瓶颈: LLM 推理 (占 99%+)
Agent 自身延迟: < 1 ms (可忽略)
```

> 系统自身的处理延迟 (安全护栏 + Tool 执行 + SSE 推送) 累计 < 1 ms，瓶颈在 LLM Provider 的推理速度，与 Agent 无关。

## 6. 压力测试边界

| 极限条件 | 行为 | 结果 |
|----------|------|:--:|
| Tool 调用超时 (>10s) | `run_command` 自动终止 | ✅ 不阻塞 |
| journalctl 不可用 | 自动降级到 dmesg / 日志文件 | ✅ 优雅降级 |
| dig / getent 不可用 | 返回空结果 | ✅ 不报错 |
| 所有 45 个 Tool 同时注册 | 38 ms 完成 | ✅ |
| 命令参数含 `rm` / `-rf` | 白名单拦截 | ✅ |
| 磁盘满 | disk_inspect 告警 | ✅ |
| 466 进程遍历 | process_tree 34 ms | ✅ |

## 7. 测试结论

| 指标 | 目标 | 实测 | 判定 |
|------|:--:|:--:|:--:|
| 安全护栏延迟 | < 1 ms | **0.026 ms** | ✅ 远超 |
| Tool 调用延迟 (只读) | < 1 s | **0.3~587 ms** | ✅ |
| 内存占用 (后端) | < 200 MB | **32 MB** | ✅ 极低 |
| 模块冷启动 | < 5 s | **38.6 ms** | ✅ |
| 并发 Tool 调用 | 不阻塞 | **正常** | ✅ |
| Tool 超时保护 | 10 s 兜底 | **生效** | ✅ |
| 命令白名单拦截 | 100% | **100%** | ✅ |

**综合评定**: SRE-agent 自身处理延迟 < 1 ms，内存占用 32 MB，安全检测零感知开销。端到端延迟瓶颈在 LLM 推理而非 Agent 本身，系统架构性能优异。
