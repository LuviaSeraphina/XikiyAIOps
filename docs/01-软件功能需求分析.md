# 软件功能需求分析

## 1. 项目背景

### 1.1 传统运维的困境

在 Linux 服务器的日常运维中，绝大多数工程师的工作模式仍然停留在「SSH + Shell + 人工」的阶段，面临以下普遍性问题：

- **故障响应高度依赖个人经验**：一位资深运维工程师可能需要数年积累才能熟练掌握系统排查的全套技能，而新人上手周期漫长
- **重复性工作占比高**：日常巡检、日志检查、磁盘清理等操作频繁重复，占用大量工作时间
- **操作安全缺乏保障**：所有操作直接在服务器上执行，无人审核、无追溯、无回滚机制
- **工具链碎片化**：监控、日志、审计、安全各自一套系统，运维工程师需要在多个平台间切换

### 1.2 需求来源

本项目来源于真实的运维场景。任何一位运维工程师在日常工作中频繁遇到以下困境：排查故障需要 SSH 登录后手动执行多条命令、重复性操作占用了大量时间、操作安全无保障、知识难以积累复用。现有的开源方案（Prometheus/Zabbix + Grafana）提供监控面板但无法执行操作，商业 AIOps 平台成本高且部署复杂，市场缺乏一款**轻量级、面向单机运维场景**的开源智能运维 Agent。

## 2. 痛点分析

### P1：人力成本高企——故障排查依赖资深工程师，新人培养周期长

**问题描述：**
一次典型故障的排查链路如下：

```
发现异常 → SSH 登录 → 翻查日志 → 搜索命令 → 
逐条执行 → 分析输出 → 对比基线 → 定位根因 →
搜索修复方案 → 执行修复 → 验证结果
```

这条链路的每一步都依赖工程师的领域知识和经验积累：

- **资深工程师 vs 新人的效率差异可达 6 倍以上**：30 分钟 vs 3 小时
- **每次故障都是一次性的**：排查思路、所用命令、分析结论全部留在工程师大脑中，无法复用
- **人员流动带来知识流失**：一位资深运维离职，带走的可能是多年积累的排查模式和故障案例库
- **团队排班成本高**：7×24 小时值班需要多人轮换，人力成本占运维总成本的 60% 以上

**实际测试对比：**

| 场景 | 手动操作耗时 | XikiyAIOps 耗时 | 效率提升 |
|------|------------|----------------|---------|
| 系统垃圾清理（磁盘清理、日志截断、coredump 清理） | 15-20 分钟 | 40 秒 | ~25 倍 |
| CPU 高负载排查（定位进程 → 分析资源占用 → 生成报告） | 15 分钟以上 | 2 分钟 | ~7.5 倍 |
| OOM 诊断（收集 dmesg → 分析 oom_score → 检查 cgroup → 关联应用） | 30 分钟起步 | 3 分钟 | ~10 倍 |

### P2：运营成本浪费——重复性运维操作缺乏自动化

**问题描述：**
运维工程师的大量工作时间消耗在重复性操作上：

| 操作类型 | 频率 | 单次耗时 | 月消耗工时 |
|---------|------|---------|----------|
| 磁盘使用率检查 | 每日 | 5 分钟 | ~2.5 人时 |
| 安全审计（登录失败、异常进程等） | 每周 | 30 分钟 | ~2 人时 |
| 系统日志排查（异常事件、服务失败等） | 按需 | 20 分钟/次 | 不定 |
| 配置巡检（Swap、挂载、服务状态等） | 每周 | 15 分钟 | ~1 人时 |

这些操作模式固定、重复性高、但必须人工执行——因为每台服务器的状态不同，无法用简单的脚本覆盖所有边界情况。这类「半自动化」工作占据了运维工程师约 40% 的工作时间。当管理多台机器时，这些时间成本线性放大。

### P3：安全风险不可控——操作无护栏、无审计、无追溯

**问题描述：**
SSH 直连服务器的运维模式存在结构性的安全缺陷：

1. **操作前无防护**：Shell 命令没有风险预检，`rm -rf /var/log/` 和 `rm -rf /` 之间只有一次键盘敲击的距离。据 Gartner 报告，人为配置错误是导致系统故障的头号原因（占 40%+）。
2. **操作中无拦截**：即便发现即将执行危险操作，Shell 本身无法在命令执行前进行拦截和确认。
3. **操作后无追溯**：谁在什么时间执行了什么命令？当时系统状态如何？执行结果怎样？传统操作审计要么未开启，要么日志量巨大难以检索，要么只记录「执行了 lsblk」这类缺乏上下文的信息。

**实际案例：**
某数据中心运维人员在执行磁盘清理时，因命令路径拼写错误，误删了相邻的业务数据目录，导致核心应用中断 4 小时。事后审计发现：
- 操作记录仅在 `.bash_history` 中留下一行无法确认上下文的命令
- 无法追溯操作时系统的完整状态
- 无法判断是误操作还是有意行为

### P4：诊断效率低——信息碎片化，MTTR 过长

**问题描述：**
系统异常的诊断过程需要从多个维度采集信息，然后人工整合分析：

| 异常现象 | 所需信息维度 | 涉及命令数 |
|---------|------------|----------|
| 磁盘写满 | 使用率、大文件、inode、文件系统类型、进程占用 | 5-8 条 |
| 内存溢出 | dmesg、oom_score、进程内存、cgroup 限额、Swap 使用 | 6-10 条 |
| CPU 高负载 | top 进程、线程级占用、中断分布、运行队列 | 5-7 条 |
| 网络延迟 | 连接数、重传率、带宽、路由、DNS | 6-10 条 |

每个场景需要 5-10 条不同的命令，输出格式各不相同，需要人工拼接理解。这导致：

- **平均故障修复时间（MTTR）以十分钟到小时计**
- **60% 的时间花在「查命令、拼信息」上，而非真正的分析和决策**
- **跨场景叠加异常（如磁盘满 + 高 CPU）诊断难度指数级上升**

### P5：工具生态碎片化——运维系统林立，操作割裂

**问题描述：**
运维工程师在管理一台服务器时，也需要同时使用多个不同的工具：

| 用途 | 典型工具 | 使用成本 |
|------|---------|---------|
| 监控告警 | Prometheus + Grafana | 只能看趋势，无法操作；每新增指标需配置 |
| 日志分析 | ELK / Loki | 查询需学习专用 DSL；无法与系统状态关联 |
| 故障排查 | SSH + Shell | 无引导、无记录、无协作 |
| 操作审计 | auditd / rsyslog | 日志量大（日均 GB 级），检索像大海捞针 |
| 安全扫描 | 手动或半自动化脚本 | 低频（月/季度），无法持续 |

这些系统之间没有数据流通，没有操作联动。运维工程师需要记忆多个系统的登录方式、查询语法和操作流程。一个典型的跨系统排查流程可能涉及：查 Grafana → 切 SSH → 翻日志 → 查 CMDB → 回 SSH 执行操作 → 回 Grafana 验证，耗时且容易遗漏关键信息。

## 3. 痛点 → 解决方案映射

| 痛点 | 解决方案 | 对应产品能力 |
|------|---------|------------|
| P1 人力成本高企 | 对话式智能运维，降低对资深工程师的依赖 | 自然语言驱动 82 个 MCP 工具、「一句话」完成复杂排查 |
| P2 重复操作成本 | 场景模板 + 自动规划执行链，一键完成标准运维流程 | 10 个预置场景模板（A-J），覆盖 80% 日常运维场景 |
| P3 安全风险 | 三层安全护栏 + 五阶段全链路审计 | 风险分级预检 + 高危操作确认弹窗 + 审计回溯 |
| P4 诊断效率 | 多信息自动聚合、一键生成诊断报告 | Agent 自动编排工具链、多维度数据关联分析 |
| P5 工具碎片化 | 统一对话入口，取代多系统切换 | MCP 插件架构集成全部运维能力，单一入口触达 |

## 5. 功能需求

### F1：自然语言驱动的系统感知

- 用户以自然语言提问，Agent 自动选择工具采集系统状态
- 支持：CPU/内存/磁盘/网络/进程/日志/容器全维度感知
- 实时流式输出，所见即所得

### F2：智能运维规划与执行

- 基于用户意图（12 种场景类型）自动编排执行计划
- 三步流水线：规划（Plan）→ 执行（Execute）→ 总结（Summarize）
- 支持多步骤、带依赖关系的任务链

### F3：安全护栏体系

- 三层风险等级：只读、受限、高危
- 高危操作弹窗确认（支持勾选/跳过/全选）
- 全量操作写入 5 阶段审计日志

### F4：RAG 知识增强

- 麒麟 V10/V11 运维知识向量化存储
- 场景模板（A-J：磁盘清理、OOM 诊断、安全审计等）
- 自动检索匹配 + 上下文注入

### F5：审计与异常回溯

- 5 阶段闭环审计（输入→感知→推理→校验→执行）
- 异常自动标记 + 因果链分析
- 同会话关联操作追溯

### F6：系统健康与告警

- 综合健康评分（0-100）
- 资源使用风险分级（安全/警告/危险）
- 安全告警聚合展示

### F7：可视化 Dashboard

- 实时系统状态卡片（CPU/内存/磁盘/网络/健康）
- 系统概览：主机名、OS、内核、启动时间
- 自动刷新（10 秒间隔）

## 6. 非功能需求

### N1：响应性能
- 工具调用延迟 < 3 秒
- Dashbaord 数据刷新间隔 ≤ 10 秒
- SSE 流式输出首包 < 1 秒

### N2：安全性
- 所有 Shell 操作经 MCP 插件层，不直接暴露终端
- 命令白名单 + 参数校验
- 高危操作二次确认

### N3：可部署性
- 全离线安装：单 .deb 包包含全部依赖
- 支持架构：x86_64 + LoongArch64

### N4：可追溯性
- 审计日志保留所有操作 5 阶段数据
- 异常操作自动标记 + 原因分类
- 审计数据支持按时间/用户/风险等级/异常类型多维筛选
